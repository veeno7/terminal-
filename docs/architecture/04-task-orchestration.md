# 04 - Task Orchestration System

CortexForge manages complexity through a hierarchical task orchestration system. It transforms vague, high-level user goals into structured, executable execution plans with robust error handling.

## The Hierarchy of Execution

### 1. The Goal
The entry point of any action.
- **Example**: "Create a blog application and deploy it."
- **Properties**: Priority, deadline, success criteria, and constraints.

### 2. The Plan (Task Dependency Graph)
A Directed Acyclic Graph (DAG) of tasks generated by the **Planner** subsystem.
- **Nodes**: Individual tasks.
- **Edges**: Dependencies (Task B cannot start until Task A completes).
- **Branching**: Support for parallel execution paths.

### 3. The Task
The smallest unit of work recognized by the Orchestrator.
- **Properties**: `id`, `description`, `status` (pending, running, completed, failed), `retry_count`, `result_data`.
- **Execution**: Linked to one or more **Skills**.

---

## Orchestration Logic

The **Task Orchestrator** is responsible for the lifecycle of the Plan:

### Scheduling
- **Topological Sort**: Determines the execution order based on dependencies.
- **Concurrency Control**: Manages how many tasks can run in parallel based on resource availability (e.g., "Don't run two git commits on the same repo at once").
- **Dynamic Re-planning**: If a task fails or reveals new information, the Orchestrator can trigger a "Re-plan" event, modifying the remaining DAG.

### Progress Tracking & Resumption
- **State Persistence**: The entire Task Graph is persisted to the database.
- **Checkpointing**: After each task completes, the agent's internal state is saved.
- **Crash Recovery**: If the system restarts, the Orchestrator reads the last state and resumes from the first `pending` task.

### Error Recovery & Retry Logic
- **Local Retry**: For transient errors (e.g., network timeout), the Orchestrator retries the task up to $N$ times.
- **Alternative Pathing**: If a task fails permanently, the Orchestrator asks the Planner for an alternative way to achieve the same sub-goal.
- **User Intervention**: If no alternative is found, the Orchestrator pauses and requests guidance from the user.

---

## The Planning Loop

1.  **Decomposition**: The LLM analyzes the Goal and generates a list of required Tasks and their dependencies.
2.  **Validation**: The Orchestrator checks if the necessary skills and resources are available.
3.  **Execution**: Tasks are fed into the **Skill Executor**.
4.  **Integration**: Results from tasks are fed back into **Working Memory**.
5.  **Completion**: Once the terminal node of the DAG is reached, the Goal is marked as "Succeeded."

## Monitoring and Introspection
The Orchestrator exposes a real-time API for the Web UI to visualize:
- The current DAG and its progress.
- Logs for each specific task.
- Resource consumption per task.
